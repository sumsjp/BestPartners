这个Agent时代最关键的接口协议

MCP的联合创造者大卫·索里亚·帕拉David Soria Parra

为什么要把它捐赠给Linux基金会

今天大飞就来给大家分享一下
大卫与主持人聊的第一个话题是
MCP究竟是从哪里冒出来的？

而搞出来的副业
MCP也不例外

地点是Anthropic位于伦敦的办公室
那时候
Anthropic内部的研究员和工程师们面临着一个巨大的痛点

Claude，但是在日常工作中
他们依然感到非常憋屈
大卫回忆，当时的情况是
所有的模型都实际上被困在一个盒子里
工程师们如果想要让Claude帮忙写代码
或者分析内部的文档
必须进行极其繁琐的复制粘贴工作
这种低效让大卫感到沮丧
当时，他接到了一个任务
要想办法让内部的研究员和工程师
能够更顺畅地在日常工作中使用Claude
他的目标是让这些最聪明的大脑
能够把他们最关心的工作流
直接与模型连接起来
当时
大卫找到了他的同事贾斯汀·斯帕-萨默斯（Justin Spar-Summers）
两人在伦敦的一间会议室里
面对着一块白板
开始构思解决方案
起初，这个项目的名字并不叫MCP
大卫最初的想法很简单
做一个名为Claude Connect的小程序
这个程序就像一个本地的代理
运行在你的电脑桌面上
一头连着Claude
另一头连着各种应用程序
但是在白板上画着画着
他们意识到不对劲了
如果只是做一个Claude Connect
那这就只是Anthropic自家的一个专有工具
这就好比是你为了给苹果手机充电
发明了Lightning接口
虽然好用，但是它只能给苹果用
如果你换了安卓手机，这根线就废了
于是，在那个会议室里
一个更宏大的想法诞生了
为什么不把这中间的连接层抽离出来
做成一个通用的标准呢？
这就好比
与其发明一万种不同的充电线
不如直接发明USB-C
于是
Claude Connect的概念被抛弃了
取而代之的是CSP（Context Server Protocol）
上下文服务器协议
也就是后来我们熟知的MCP的前身
这一刻
它标志着这个项目从一个产品功能
升维成了一个行业标准
那么，这个所谓的MCP
在技术上到底是如何运作的？
为什么大卫和主持人斯图尔特·里奇（Stuart Ritchie）
都喜欢用USB-C来打比方呢？
为了讲清楚这个技术原理
我们需要回顾一下计算机接口的发展史
如果你是像大飞我这样
是个经历过90年代计算机的老登
一定记得当年的电脑背面是什么样子
那简直是一场噩梦
打印机要用巨大的并口（Parallel Port）
鼠标键盘要用PS/2接口
外置猫要用串口（Serial Port）
游戏手柄还有专门的游戏接口
每一种设备都有自己独特的物理形状、针脚定义和通信协议
这就导致了极高的连接成本
如果你买了一个新的设备
你不仅要担心插不插得上
还要安装专门的驱动程序
而现在的AI生态
其实就处于那个接口大乱炖的90年代
假设你是一家开发AI笔记应用的公司
你想让你的AI能读取用户的谷歌日历
那你需要专门写一套代码来对接谷歌的API
下个月，用户说我还想连接Notion
那你又得去读Notion的文档
再写一套完全不同的代码
再下个月，用户说我用的是Outlook
你还得继续写
这就是开发者们最头疼的NxM问题
如果有 N个模型
想要连接 M个数据源
那么整个行业就需要编写 NxM个集成插件
这是一种指数级的资源浪费
MCP的出现
就是为了把这个乘法变成加法
MCP定义了一种标准化的通用语言
它引入了三个核心角色
第一是主机，Host
也就是AI客户端
比如Claude Desktop、Cursor编辑器或者VS Code
第二是服务器，Server
这是一个轻量级的程序
它负责与特定的数据源
比如Postgres数据库、Slack、Google Drive等等打交道
第三是客户端，Client
它是主机内部用来和服务器对话的组件
在这个架构下
MCP协议就像是USB-C的那根线
作为AI模型
我不需要知道你是Postgres数据库还是Excel表格
我只需要通过MCP这个标准接口访问你
让你把里面的数据结构告诉我
大卫在访谈中用了一个非常形象的比喻
MCP就是给拥有强大大脑的模型
装上了通用的四肢
在此之前
开发者不得不为每一个模型提供商
一遍又一遍地编写相同的集成代码
这不仅枯燥，而且极其低效
通过MCP
任何人只需要编写一次针对某个数据源的MCP Server
那么理论上
任何支持MCP协议的AI客户端
都可以直接连接并且使用这个数据源
这就是标准化的力量
它消除了中间的摩擦力
让数据和智能能够无缝流动
但是这引出了一个非常关键的商业问题
也是主持人斯图尔特在访谈中
直接向大卫抛出的尖锐问题
既然这个技术这么好
为什么Anthropic不把它私有化
为什么不把它作为Claude的独家护城河呢？
这是一个典型的商业博弈
在科技圈
巨头们往往倾向于构建封闭的围墙花园
苹果的生态就是最成功的例子
如果Anthropic把MCP做成私有协议
那意味着所有想要连接这些丰富数据的用户
都必须使用Claude
这听起来是一个完美的商业闭环
但是Anthropic选择了另一条路
大卫解释说
这里面有着更深层的战略考量
首先，这是一个创新者的窘境
如果Anthropic试图通过私有连接器来垄断市场
虽然短期内能留住用户
但是长远来看
他们不可能凭借一己之力
开发出连接世界上所有软件的适配器
世界上的软件数以百万计
Anthropic的工程师再多也写不完这些驱动
只有开放
只有让全球的开发者社区都参与进来
这个连接器的数量才能呈现爆炸式增长
其次，也是最重要的一点
那就是信任与中立
设想一下
如果这个标准完全掌握在Anthropic手中
其他的竞争对手
比如说OpenAI或者Google
会愿意采用这个标准吗？
显然不会

或者收取授权费
搞卡脖子那一套
所以

它必须是中立的

js等

Anthropic实际上是交出了这个项目的控制权
包括商标、代码的所有权、未来的开发决策权
都移交给了基金会
这就给所有的参与者吃了一颗定心丸
这次捐赠的规格也是非常的高

除了Anthropic

但是在MCP这件事上
他们坐在了同一张圆桌旁
说明行业巨头们已经达成了一个共识
AI Agent的互联互通时代已经到来
大家需要一条共同修建的高速公路
而不是各自为战的泥泞小道
大卫在访谈中也提到了一个有趣的细节
这让他想起了当年的Open Science运动和arXiv预印本平台
在过去，科学研究是封闭的

但是arXiv出现后

不需要谁来批准，也不需要付费
这种开放透明的机制
极大地加速了科学的迭代

当一个工具足够好用
而且足够开放的时候
它就会像水一样流进每一个缝隙
当然
任何新技术的诞生都不是一帆风顺的
在访谈中
斯图尔特和大卫也毫不避讳地谈到了
MCP面临的技术挑战和未来的演进方向
首先是安全性问题
这可能是大家最担心的一个方面
以前的AI只是陪你聊天
最多说说胡话
但是现在的MCP
是让AI连接你的真实数据和工具
那大家就会想
如果给Claude装了一个银行账户的MCP Server
它会不会在我想查余额的时候
自作主张把钱转走了呢？
这种风险被称为提示注入攻击
恶意的攻击者可能会在看似无害的文本中隐藏指令
诱骗AI去执行危险操作
大卫坦言，MCP本身作为一个通信协议
它就像是一根导线
导线本身并不负责判断流过的是电流还是毒药
但是，MCP社区正在构建一套防御机制
目前的做法是人在回路（Human-in-the-loop）
当模型试图调用一个敏感的工具时
MCP主机就会弹出一个确认框
明确的告诉用户
模型想要执行这个操作，是否允许？
这就像是我们手机APP
在申请读取相册权限时的弹窗一样
此外
MCP协议还支持更细粒度的权限控制
比如只读权限、读写权限的分离
未来，这种安全机制会更加智能化
第二个挑战是上下文膨胀（Context Bloat）
我们知道
大模型的上下文窗口虽然越来越大
但也是有限的
而且每多一个token都是成本
如果我们给AI装了50个工具
每个工具都有详细的说明文档
那么光是把这些工具的说明塞进提示词里
可能就占满了上下文窗口，不仅费钱
还会让模型注意力涣散
不知道该用哪个
针对这个问题
Anthropic正在开发一种工具搜索（Tool Search）的机制
这就像是给模型配了一个应用商店
或者搜索引擎
模型不需要一开始就加载所有工具的说明书
当用户提出一个需求后
模型会先去MCP的注册表里搜索相关的工具
找到合适的工具后
再去动态地加载这个工具的定义
这就把死记硬背变成了现查现用
大大提高了效率
第三个有趣的技术点是关于有状态（Stateful）的讨论
如果你熟悉Web开发的话
就知道HTTP协议是无状态的
你发一个请求，服务器回一个响应
然后它就忘了你是谁
但是MCP的设计是有状态的
大卫解释说
这是因为AI Agent的行为往往是连续的
比如，你在调试代码
这是一个持续的过程
模型需要知道你上一步做了什么
需要保持与调试器的长连接
MCP建立在JSON-RPC之上
维护了一个长期的会话
这也为未来更复杂的Agent-to-Agent通信打下了基础
在访谈的后半段
大卫描绘了一副非常有意思的未来
现在的MCP主要还是解决
人指挥AI用工具的问题
但是下一步，我们将看到AI指挥AI
想象一下
你有一个负责写代码的AI Agent
还有一个负责做安全测试的AI Agent
通过MCP，这两个Agent可以互相通信
写代码的Agent写完一段函数
通过MCP发给测试Agent
测试Agent跑完测试
把报错信息通过MCP回传给写代码的Agent
它们可以自己在后台协作
直到任务完成
最后只给你一个结果
这也就是为什么Linux基金会的新组织叫Agentic AI Foundation
这里的Agentic指的就是这种具有自主性、能规划长程任务的能力
为了支持这种能力
MCP协议正在引入任务（Tasks）的概念
未来的AI不仅仅是回答你的问题
而是可以接收一个像深度调研某个行业这样的复杂指令
然后在后台运行几个小时
期间不断调用各种MCP工具去搜集信息、整理数据
最终生成报告
除此以外
应用层的形态也会发生巨变
目前的AI交互主要还是基于聊天框
但是在某些场景下
聊天并不是最高效的
比如你想选一个飞机座位
或者在地图上圈一块地
因此
Anthropic正在探索一种名为MCP Apps的概念
这就是让MCP服务器不仅仅返回文本数据
还能返回UI组件
举个例子
当你让Claude帮你订机票的时候
航空公司提供的MCP Server
可以直接在Claude的界面里渲染出一个座位选择图
你可以直接点击选座
而不是用文字说我要第3排A座
这让AI的交互界面从纯文本
进化到了富媒体交互
这将是一个巨大的体验飞跃
在视频的最后
我想聊聊这对我们普通人意味着什么
可能很多观众觉得
这不就是程序员折腾的底层协议吗？
跟我有什么关系？

AI可以直接调用你公司的内部数据库
自动填充数据

也能通过MCP理解你复杂的模糊指令
