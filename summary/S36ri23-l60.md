## Agent Harness：未来AI竞争的核心博弈

大家好，我是大飞，欢迎来到最佳拍档。今天，我们将深入探讨一个在全球AI顶流团队（如Anthropic、OpenAI、LangChain、Perplexity）中引起疯狂加码的焦点领域——Agent Harness。甚至有人断言，未来AI的竞争，将不再是模型参数的内卷，而是Harness架构的博弈。

### 为什么需要Agent Harness？

当前行业普遍存在一个痛点：许多智能体在演示环境中表现流畅，一旦投入生产环境，任务成功率便会断崖式下跌，立刻“原形毕露”。多数开发者会下意识地归咎于“模型不够强大”，但事实并非如此。

**真相是：问题症结不在模型本身，而在于包裹模型的那套基础设施。**

LangChain的一项实验有力地证明了这一点：在不改动模型权重和底层算法的前提下，仅仅优化了包裹大语言模型的Harness架构，就让智能体在TerminalBench 2.0评测中，从30名开外跃升至第5名。更有研究团队让LLM自主优化Harness架构，任务通过率飙升至76.4%，超越所有人工设计的系统。这就是Agent Harness的强大威力。

### 什么是Agent Harness？

“Agent Harness”这个术语在2026年初被全球AI社区正式定义，但其理念早已渗透在每一个生产级AI应用之中。

**简而言之，Agent Harness就是包裹大语言模型的一整套操作系统级软件基础设施。** 它将一个仅能输出文本、无状态、容易出错的“裸大语言模型”，转化为一个有目标、会使用工具、能自我纠错、可持久运行的可靠智能体。

LangChain的维韦克·特里维迪（Vivek Trivedy）曾指出：“如果你不是模型，你就是Harness。”这句话道出了其核心本质：构建智能体，本质上是搭建一套Harness，并将其对接给模型。

我们可以用计算机架构做一个类比：
*   **裸大语言模型**：就像一台没有内存、硬盘和外设驱动的CPU，仅有核心计算能力，无法独立完成实际任务。
*   **上下文窗口**：充当临时内存，速度快但容量有限。
*   **向量数据库与长期存储**：充当硬盘，容量大但响应慢。
*   **工具集成**：就是设备驱动，让模型能调用外部能力。
*   **Agent Harness**：就是让这一切协同工作的“操作系统”。

贝伦·米利奇（Beren Millidge）在2023年的《AI的脚手架》一文中更直言，我们通过Agent Harness，“重新发明了冯·诺依曼架构”。这并非简单的技术封装，而是计算系统发展的一种必然抽象，是任何AI智能体走向实用所必须的底层逻辑。

### Harness、提示工程与上下文工程的层次

在深入拆解Harness之前，我们需要理清三个容易混淆的工程层级：

1.  **提示工程 (Prompt Engineering)**：专注于打磨模型接收的指令，确保模型更精准地理解需求。
2.  **上下文工程 (Context Engineering)**：核心是管理模型在不同阶段能看到哪些信息，避免信息过载。
3.  **Harness工程 (Harness Engineering)**：涵盖了前两者，更囊括了工具编排、状态持久化、错误恢复、验证循环、安全管控、生命周期管理等全套应用基础设施。

**Harness绝非仅仅是给提示词套个壳。** 它是一套让自主智能体实现自主思考、自主行动、自主修复的完整系统，是“玩具级演示”与“生产级智能体”之间的本质区别。

### Agent Harness的十二大核心模块

综合Anthropic、OpenAI、LangChain以及全球AI工程社区的最佳实践，一个真正能落地的生产级Agent Harness，需要由十二个完全独立、环环相扣的核心模块组成。少了任何一个，都难以支撑稳定的智能体运行。

1.  **编排循环 (Orchestration Loop)**
    *   **核心功能**：智能体的心跳，所有行为的核心引擎。常见的ReAct循环、思考-行动-观察（TAO）循环都是其具体实现。
    *   **运行逻辑**：组装提示词 -> 发送给大模型 -> 解析输出 -> 判断并调用工具 -> 执行工具 -> 返回结果 -> 循环。
    *   **设计理念**：Anthropic称之为“笨循环”，即智能决策由模型完成，Harness只负责按流程转场、调度任务，不参与核心推理。

2.  **工具 (Tools)**
    *   **核心功能**：智能体的“手”，与现实世界交互的唯一途径。以标准化Schema形式定义的能力集合。
    *   **职责**：工具注册、Schema校验、参数提取、沙箱执行、结果捕获，并将结果格式化回传。
    *   **能力边界**：工具层的设计直接决定了智能体的能力边界。头部厂商如Anthropic Claude Code、OpenAI Agents SDK都构建了完善的工具体系。

3.  **记忆 (Memory)**
    *   **核心功能**：智能体跨越时间尺度、保持任务连续性的关键，避免“鱼的记忆”。
    *   **记忆类型**：
        *   **短期记忆**：单次会话内的对话历史。
        *   **长期记忆**：跨会话持久化，即便重启也能保留任务进度、决策记录。
    *   **实现方式**：Anthropic通过`.md`文件，LangGraph采用JSON存储，OpenAI支持SQLite或Redis。Claude Code更是设计了三级记忆层级（轻量级索引、详细主题文件、原始交互记录）。
    *   **设计原则**：智能体将记忆作为一种提示，行动前会与实际状态核对验证，避免因记忆错误导致任务失败。

4.  **上下文管理 (Context Management)**
    *   **核心功能**：解决生产级智能体“上下文腐烂（Context Rot）”的问题，确保模型性能。
    *   **痛点**：当关键信息落在上下文窗口中间时，模型性能会暴跌30%以上，且随着内容膨胀，指令遵循能力和推理准确率会持续下降。
    *   **应对策略**：
        *   **压缩 (Compaction)**：对话历史摘要，保留核心决策，丢弃冗余工具输出。
        *   **观察屏蔽 (Observation Masking)**：隐藏旧工具输出细节，保留调用记录。
        *   **即时检索 (Just-in-time Retrieval)**：维护轻量级索引，动态加载所需数据（如Claude Code的`grep`、`glob`）。
        *   **子智能体委派 (Sub-agent Delegation)**：任务拆分给子智能体，只返回精简摘要。
    *   **核心准则**：找到“最小的高信噪比Token集合”，用最少的关键信息最大化任务效果。

5.  **提示词组装 (Prompt Assembly)**
    *   **核心功能**：定义模型在每一轮推理中“看到的世界”，连接上下文、记忆、工具和用户需求。
    *   **组装过程**：分层堆叠的结构化过程，优先级明确。
    *   **标准顺序**：系统提示词（身份规则）→ 工具定义（可用能力）→ 记忆文件（历史经验）→ 对话历史（任务进度）→ 用户消息（最新需求）。
    *   **OpenAI Codex**：采用严格的优先级栈，确保核心规则不被覆盖。

6.  **工具调用与结构化输出 (Tool Calling & Structured Output)**
    *   **核心功能**：模型与Harness之间的通用语言，解决传统自由文本输出难以解析的问题。
    *   **现代化Harness**：完全依赖原生工具调用，模型直接返回标准化的`tool_calls`结构化对象。
    *   **Harness判断逻辑**：解析模型输出，存在工具调用则执行并循环，否则将模型输出作为最终答案终止循环。
    *   **结构化输出**：OpenAI和LangChain支持Pydantic框架进行Schema约束，降低解析失败率。

7.  **状态与检查点 (State & Checkpointing)**
    *   **核心功能**：实现智能体断点续跑、可回溯和可调试，解决长周期任务中断后无法恢复的痛点。
    *   **长流程任务**：如代码重构、数据分析，可能持续数小时甚至数天，状态保存至关重要。
    *   **实现方式**：LangGraph将状态建模为类型化字典，检查点在超级步骤边界触发。OpenAI提供四种互斥状态策略。Claude Code则用Git提交作为检查点，进度文件作为结构化草稿本。

8.  **错误处理 (Error Handling)**
    *   **核心功能**：智能体在复杂环境中稳定运行的安全网。
    *   **挑战**：一个10步任务，每步99%成功率，总成功率仅约90.4%，错误会滚雪球般放大。
    *   **分类与处理 (LangGraph)**：
        *   **瞬时错误**：网络波动、API限流 → 带退避策略的重试。
        *   **模型可恢复错误**：参数错误、逻辑失误 → 包装成工具消息返回给模型自主调整。
        *   **用户可修复错误**：权限不足、配置错误 → 中断流程等待人工输入。
        *   **意外错误**：系统崩溃、底层异常 → 直接抛出，便于调试。
    *   **其他策略**：Anthropic在工具处理器内部捕获所有失败，确保主编排循环不中断。Stripe严格限制重试次数。

9.  **护栏 (Guardrails)**
    *   **核心功能**：智能体的安全红线，防止智能体做出越权、有害、违规操作，是企业级应用的核心保障。
    *   **OpenAI三层防护**：
        *   **输入护栏**：过滤恶意和违规输入。
        *   **输出护栏**：确保最终输出内容合规安全。
        *   **工具护栏**：管控工具调用权限，防止高风险操作。
    *   **Anthropic设计**：将权限执行与模型推理完全解耦，模型只思考，工具系统判断能否做。Claude Code可独立管控40种工具能力，分阶段（加载、调用前、高风险操作）严格把关。

10. **验证与反馈 (Verification & Feedback)**
    *   **核心功能**：区分玩具级智能体与生产级智能体，没有验证的智能体，输出结果永远不可信。
    *   **三种通用验证方式 (Anthropic)**：
        *   **基于规则的反馈**：测试用例、Linter、类型检查器等确定性工具。
        *   **视觉反馈**：Playwright等工具截图，检查UI任务完成效果。
        *   **模型当裁判**：独立的子智能体评估主智能体输出，给出语义、逻辑反馈。
    *   **重要性**：加入验证机制，能让输出质量提升2到3倍，是保证智能体产出价值的必要投入。

11. **子Agent编排 (Subagent Orchestration)**
    *   **核心功能**：让单个智能体升级为智能体集群，解决复杂、大规模、多领域的任务需求。
    *   **应用场景**：当任务涉及多个专业领域、工具过多、流程复杂时。
    *   **主流实现方案**：
        *   **Claude Code**：支持Fork、Teammate、Worktree三种执行模式。
        *   **OpenAI SDK**：支持Agents-as-tools（专家Agent处理细分任务）和Handoffs（任务全面交接）模式。
        *   **LangGraph**：将子Agent实现为嵌套状态图，通过图结构管理任务流转。

12. **初始化与环境搭建 (Initialization & Environment Setup)**
    *   **核心功能**：所有模块协同工作的起点，定义智能体从启动到运行的完整生命周期。
    *   **标准执行周期**：
        1.  **提示词组装**：整合所有信息构建完整输入。
        2.  **模型推理**：发送提示词，生成文本或工具调用。
        3.  **输出分类**：判断工具调用、任务交接或直接输出。
        4.  **工具执行**：校验、权限检查、沙箱运行（读并发，写串行）。
        5.  **结果打包**：格式化工具结果和错误信息。
        6.  **上下文更新**：追加结果到对话历史，触发上下文压缩。
        7.  **循环执行**：回到第一步，直到满足终止条件。
    *   **终止条件**：模型无工具调用、达到最大轮次、Token预算耗尽、护栏触发、用户中断、安全拒绝等。

### 全球主流智能体框架的Agent Harness实践

理解了Harness的模块，我们再来看看主流框架如何落地：

*   **Anthropic Claude Agent SDK**：
    *   **哲学**：薄Harness哲学的极致代表，信任模型、简化框架。
    *   **特点**：极简“笨循环”，遵循Gather-Act-Verify流程，Git提交做状态管理。
    *   **优势**：轻量、高效，与模型深度耦合。
*   **OpenAI Agents SDK**：
    *   **哲学**：代码优先设计理念。
    *   **特点**：Runner类实现Harness，支持异步/同步/流式，工作流用原生Python编写。
    *   **优势**：开发者友好、快速落地生产级应用。
*   **LangGraph**：
    *   **哲学**：从LangChain进化，采用图结构设计。
    *   **特点**：Harness建模为显式状态图，通过节点和条件边控制流程，支持嵌套状态图。
    *   **优势**：显式流程控制、可调试性强，适合复杂、多分支的长流程任务。
*   **CrewAI**：
    *   **哲学**：主打基于角色的多智能体架构。
    *   **特点**：将智能体、任务、团队解耦，通过Flows层实现路由与验证。
    *   **优势**：多智能体协作、角色分工明确，适合团队式、多角色的复杂任务场景。
*   **AutoGen (微软智能体框架)**：
    *   **哲学**：开创对话驱动编排先河。
    *   **特点**：支持顺序、并发、群组聊天、交接、magentic五种编排模式，将对话作为协作协议。
    *   **优势**：适合开放式、多智能体交互的场景。

这些框架的核心模式高度趋同，都围绕编排循环、工具、记忆、上下文等核心模块构建，但设计哲学截然不同。没有绝对的优劣，只有场景的适配。

### 脚手架隐喻与共同进化

我们将Harness比作建筑上的**脚手架**——它是让模型落地为智能体的临时支撑。随着模型能力的增强，Harness的复杂度应该越低。行业实践已验证：Manus项目半年内重构五次，每次做减法，性能反而持续提升。

这背后是**共同进化原则**：现代大模型在后训练阶段会将特定Harness纳入训练循环，模型与框架深度耦合。模型内化的能力越多，Harness需要的封装就越少。

一个优秀的Harness设计，必须能通过“面向未来的测试”：即模型升级后，智能体性能自然提升，而无需增加Harness复杂度。未来的行业趋势一定是：**更薄的Harness、更强的模型、更模块化的架构。**

### 构建Agent Harness的七大架构抉择

所有AI工程师在搭建Agent Harness时，都必须面对以下七大核心考题：

1.  **单智能体 vs. 多智能体**：行业共识是先榨干单智能体性能。仅当工具重叠超10个或任务域明显分离时，才考虑拆分多智能体。
2.  **ReAct循环 vs. 计划-执行循环**：
    *   ReAct：灵活，但每步成本高。
    *   计划-执行：分离规划与执行，LLMCompiler数据显示比顺序ReAct快3.6倍。
3.  **上下文管理策略**：包括基于时间清理、对话摘要、观察掩码、结构化笔记、子智能体委派五种方法。核心是保留推理痕迹，减少Token消耗。
4.  **验证循环设计**：
    *   **计算式验证**：通过测试、Linter提供确定性。
    *   **推理验证**：让模型做裁判解决语义问题。两者结合最优。
5.  **权限与安全**：
    *   **宽松模式**：高效但有风险。
    *   **严格模式**：安全但低效。需根据部署场景平衡。
6.  **工具范围**：工具越多性能越差。Vercel砍掉80%工具后性能反而提升。原则是只暴露当前步骤所需的最小工具集。
7.  **Harness的厚度**：
    *   **薄Harness**：需要信任模型。
    *   **厚Harness**：需要编码来控制逻辑。模型越强，越应该偏向薄Harness。

### 总结

回到我们最初的问题：为什么有的智能体演示时十分流畅，生产环境却“翻车”严重？答案从来不是模型不行，而是Harness的架构可能不够完善。两个一模一样的模型，仅仅因为Harness设计不同，性能可能就会天差地别。

2026年的AI竞争，早已不是单纯模型参数的内卷，而是Harness工程的较量。如何把上下文当稀缺资源管理、如何设计拦截错误的验证循环、如何构建无幻觉的记忆系统、如何平衡脚手架与模型的能力——这才是AI工程化的核心硬骨头。
